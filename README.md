# LAWNET — Bench Assistant

**Legal Assessment and Verification Network**
A citation integrity and verification tool for Research Assistants at the Nigerian Court of Appeal. Designed to detect unverifiable, fabricated, or legally displaced authorities before files reach a Justice's desk.

> **Status:** Prototype — active pilot development. Targeting a 6-month controlled deployment at the Court of Appeal, Abuja Division.

---

## Overview

LAWNET is a browser-based bench tool built for court Research Assistants. It does not write judgments, draft opinions, or make legal determinations. Its sole function is citation verification — extracting case law references and statutory provisions from submitted briefs, checking them against a seeded Nigerian caselaw registry, and flagging anything that cannot be matched before it enters the judicial workflow.

The product addresses a specific and growing operational risk: the submission of unverified or fabricated legal citations generated by AI writing tools, which cannot be detected by standard manual review at scale.

**Core capabilities:**

- Extract statutory and case law citations from pasted brief text
- Verify each citation against a registry seeded from LPELR, NWLR, and Court of Appeal records
- Flag unverifiable citations, overturned authorities, and distinguished precedents
- Generate annotated brief views and bench memo drafts for Research Assistant review
- Map precedent relationships through a relational inference knowledge graph
- Maintain a read-only audit log of all actions taken on each case file

## Application Architecture

The prototype is structured as a single-page application shell with tab-based navigation. All state is held in memory; there is no persistence layer in the prototype.

### Shell layout

```
┌─────────────────────────────────────────────────────────────┐
│  TOPBAR  [Seal] LAW-NET · Case ref · Status · Actions       │
├─────────────────────────────────────────────────────────────┤
│  TAB BAR  Case Review │ Citation Validator │ Knowledge Graph │
│           Pipeline Info │ Audit Log                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  CONTENT AREA  (per-tab page panels)                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Tab pages

| Tab | Description |
|---|---|
| **Case Review** | Workflow progress strip, case-level risk metrics, document chip summary, annotated brief view, bench memo draft, and full authority review table |
| **Citation Validator** | Paste-and-verify interface; animated scan sequence; per-citation result cards with status, court, year, and action guidance; summary bar |
| **Knowledge Graph** | SVG-rendered precedent graph showing citation relationships, registry verification status, and appellate treatment (cited / distinguished / unverifiable edges) |
| **Pipeline Info** | Four-stage pipeline architecture reference; data source status matrix with pilot access dependencies |
| **Audit Log** | Read-only timestamped record of all actions taken on the active case file |

### Design system

The visual identity is derived from the Nigerian federal judiciary colour palette:

| Token | Hex | Usage |
|---|---|---|
| `--jg` | `#1B4332` | Primary — judiciary forest green |
| `--jg-mid` | `#2D6A4F` | Hover states, secondary surfaces |
| `--gold` | `#B7862A` | Accent — judicial gold |
| `--gold-light` | `#D4A853` | Hover gold, secondary accents |
| `--cream` | `#FAFAF8` | Page background |

Status colours follow a consistent semantic convention across all components (pills, cards, graph nodes, table rows): green for verified, red for flagged/unverifiable, amber for needs review, blue for informational.

### Knowledge graph

The graph is rendered as an inline SVG within a fixed `800×480` viewBox. Nodes represent authorities (statutes, case law, current matter); edges represent citation relationships. Edge types are visually differentiated:

- **Solid green** — cited and verified
- **Dashed red** — unverifiable citation
- **Dashed amber** — distinguished in subsequent treatment
- **Dashed grey** — statutory application

Node click events dispatch tooltip positioning relative to the viewport, with a 4-second auto-dismiss.

## Citation Verification Pipeline

The verification logic follows four sequential stages:

**Stage 1 — Ingestion & normalisation**
Submitted text is stripped of metadata and boilerplate. Citation string formats are normalised across NWLR, LPELR, and FWLR variants to produce a consistent extraction target.

**Stage 2 — Entity extraction**
NLP models tuned to Nigerian legal syntax isolate statutory provisions (Act name + section) and case law markers (case name + volume + part + page). Each extracted entity is queued for registry lookup.

**Stage 3 — Registry verification**
Each citation is checked against the seeded registry via fuzzy-match on case name, volume, part, and page. A citation must match on all four parameters to receive a verified status. Partial or absent matches are flagged immediately for Research Assistant review rather than passed through.

**Stage 4 — Relational inference**
Verified citations are mapped against a knowledge graph of established appellate lines. The graph exposes overturned authorities (where a Supreme Court decision has displaced the relied-upon proposition) and distinguished authorities (where the proposition has been narrowed in subsequent treatment).

### Registry data sources — Phase 1 pilot scope

| Source | Coverage | Pilot status |
|---|---|---|
| LPELR (LawPavilion Electronic Law Reports) | Structured digital format | Available |
| NWLR (Nigerian Weekly Law Reports) | Print indices | Digitisation required |
| Court of Appeal internal records | Abuja Division judgment repository | Access-gated — subject to Court approval |
| NJC repository | National Judicial Council | Partial coverage |

> **Infrastructure dependency:** No comprehensive structured national caselaw database currently exists in Nigeria. Pilot deployment is contingent on the Court's data access approval for registry seeding. This is the primary precondition for moving from prototype to live deployment.

## Governance & Security Posture

LAWNET is designed for on-premise or offline deployment within the Court's controlled infrastructure. The following constraints are non-negotiable and must be preserved in any production implementation:

- **No external AI processing** — court documents and case data must not be transmitted to third-party AI services or cloud APIs
- **On-premise registry** — the verification registry runs on Court-controlled infrastructure; no caselaw data leaves the deployment environment
- **Audit logging** — every action taken on a case file (upload, verification run, memo draft, send to Justice review) is logged with timestamp and user role
- **No judgment drafting** — the system writes no legal text and makes no legal determinations; output is always a flagged citation list for human review
- **Role separation** — the system surfaces outputs to Research Assistants; Justice review is a human-initiated step outside the tool's scope
